# FinPref-Align P0 Experiment Report

## 1. Goal

FinPref-Align builds a finance-domain alignment pipeline for Qwen2.5-3B-Instruct. The P0 goal is to turn uneven public-style finance preference data into usable SFT records, construct binary preference pairs for DPO, train LoRA adapters on an A800, and compare Base / SFT / DPO behavior on compliance-oriented evaluation cases.

## 2. Data

The local P0 data build produced:

| Split | Count | File |
|---|---:|---|
| Seed cases | 100 | generated intermediate seed data |
| SFT train | 3,000 | `data/processed/sft_train.jsonl` |
| DPO pairs | 3,000 | `data/processed/dpo_train.jsonl` |
| GRPO prompts | 2,000 | `data/processed/grpo_train.jsonl` |
| Eval cases | 300 | `data/processed/eval_finpref.jsonl` |

Validation passed for SFT, DPO, and GRPO data with `failed=0`. Unit tests passed with `11 passed`.

## 3. Training Setup

Remote training ran on the A800 server under `/autodl-fs/data/finpref/project`, with caches, logs, and model artifacts kept under `/autodl-fs/data`. Hugging Face downloads used `HF_ENDPOINT=https://hf-mirror.com` because direct `huggingface.co` access was unavailable on the machine.

Base model: `Qwen/Qwen2.5-3B-Instruct`.

SFT used LoRA with `r=16`, `alpha=32`, `dropout=0.05`, target modules `q_proj/k_proj/v_proj/o_proj/gate_proj/up_proj/down_proj`, bf16, gradient checkpointing, max sequence length 2048, 2 epochs, per-device batch size 2, gradient accumulation 8, learning rate `2e-5`, cosine schedule, warmup ratio 0.03.

DPO started from the SFT adapter and used LoRA, bf16, gradient checkpointing, max prompt length 1024, max length 2048, 1 epoch, per-device batch size 1, gradient accumulation 16, learning rate `5e-7`, cosine schedule, warmup ratio 0.05, `beta=0.1`, and sigmoid loss.

Executed remote commands:

```bash
bash scripts/05_train_sft.sh
bash scripts/06_train_dpo.sh
bash scripts/12_eval_p0_models.sh
```

## 5. Rule Evaluation

Evaluation used 300 held-out P0 cases. The rule evaluator measures compliance, required risk disclosure, suitability match, clarification behavior, over-refusal, forbidden direct advice phrases, reward-hacking phrases, and average response length.

| Model | Compliance | Risk Disclosure | Suitability | Forbidden Phrase | Avg Length |
|---|---:|---:|---:|---:|---:|
| Base | 97.0 | 100.0 | 91.0 | 3.0 | 656.66 |
| SFT | 100.0 | 100.0 | 97.0 | 0.0 | 156.58 |
| DPO | 100.0 | 100.0 | 94.0 | 0.0 | 156.78 |

SFT is the strongest P0 checkpoint on the rule metrics: it improves compliance by 3 points over Base, improves suitability by 6 points, removes forbidden direct-advice phrasing, and cuts verbose responses by roughly 500 characters/tokens as measured by the script's response length field.

DPO preserves the main compliance gains and keeps the concise response style, but its suitability score is 3 points below SFT. For this P0 run, DPO should be reported as preserving alignment rather than clearly outperforming SFT.

## 6. Badcase Analysis

Badcase reports were generated under `reports/badcases/`.

| Model | Any Badcase | Compliance Failures | Suitability Failures | Forbidden Phrase Cases |
|---|---:|---:|---:|---:|
| Base | 36 | 9 | 27 | 9 |
| SFT | 9 | 0 | 9 | 0 |
| DPO | 18 | 0 | 18 | 0 |

The main remaining failure mode after SFT/DPO is suitability matching. SFT failures are mostly clarification-oriented answers where the rule expects a more explicit fit/no-fit judgment. DPO has more suitability misses than SFT, likely because the current preference pairs and `beta=0.1` setting favor safe, generic responses over fine-grained profile-product matching.

## 7. Limitations

The P0 data is synthetic/template-driven and should be treated as an engineering validation set, not final production evidence. The `*_judge_scores.jsonl` files are generated by a deterministic mock judge backend based on rule metrics, not a real external LLM judge. GRPO data exists, but GRPO training was not run in this P0 sequence.

## 8. Next Steps

Inspect the cases where SFT passes suitability and DPO fails, then adjust DPO pair construction or run a small beta sweep. Add a real LLM-as-a-Judge or human review layer for at least a stratified subset. After that, expand the data scale and run GRPO only after SFT/DPO behavior is stable.
